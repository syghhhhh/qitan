# 三、为什么不能直接一个 Prompt 端到端出结果

如果你现在直接做成：

- 输入企业名
- 调一次大模型
- 输出整份背调 JSON

会有几个问题：

## 1. 证据不足
你要求输出里有：

- recent_developments
- evidence_references
- risk_signals
- demand_signals

这些内容如果没有真实采集流程，模型只能“脑补”，可靠性会很差。

## 2. 结构化字段很多
你的输出里既有：

- 事实类字段（公司名、行业、主营业务）
- 事件类字段（融资、招聘、扩张）
- 推断类字段（需求信号）
- 决策类字段（P1/P2/P3/discard）
- 生成类字段（微信/电话/邮件话术）

这些字段的生成方式完全不同，不能混成一步做。

## 3. 商务判断需要可解释
如果销售问：

- 为什么判定为 P1？
- 为什么推荐找 IT 负责人？
- 为什么说它可能有 CRM 需求？

你必须能回溯到证据。  
所以你的系统必须是：

> **先证据，后推断；先事实，后判断；先结构化中间结果，后成品输出。**

# 四、建议的总体架构：分成 6 层

我建议把 backend 的真实智能体工作流拆成 **6 层**：

---

## 第 1 层：请求编排层（Orchestration Layer）

负责：

- 接收 `/analyze` 请求
- 标准化输入
- 创建分析任务上下文
- 调度后续各阶段
- 聚合结果
- 输出最终 `DueDiligenceOutput`

这一层相当于总控。

你可以理解为一个：

- `analyzer_service`
- 或 `analysis_orchestrator`

它不负责“具体分析”，只负责“流程推进”。

---

## 第 2 层：数据采集层（Collection Layer）

负责去收集跟企业有关的原始信息。

未来真实可接的来源包括：

- 官网
- 新闻
- 招聘站
- 企业信息平台
- 公开工商/司法信息
- 社交平台 / 公众号 / 媒体报道
- 行业目录 / 产品页 / 招股书 / 融资稿

这一层先不必一步到位全做完，但架构上一定要独立出来。

### 这一层输出的不是最终报告，而是：
- 原始文档列表
- 原始片段列表
- 每条内容的来源、时间、URL、抓取时间

比如统一成类似：

- `RawDocument`
- `EvidenceItem`
- `SourceRecord`

---

## 第 3 层：信息抽取层（Extraction Layer）

把采集回来的原始网页、新闻、文本，抽成结构化信息。

例如从原始材料中抽出：

- 企业基本信息
- 近期事件
- 招聘信息
- 产品线索
- 技术栈线索
- 扩张迹象
- 风险事件
- 组织角色信息

这一步建议大量用 LLM，但要有 schema 约束。

### 这一层的本质是：
> 从“文本证据”变成“结构化候选事实”。

不是直接下结论，而是先提炼事实。

---

## 第 4 层：分析推断层（Reasoning / Assessment Layer）

这一层才开始做“智能体分析”。

基于抽取出的结构化事实，进行：

- 企业画像归纳
- 近期动态归类
- 需求信号识别
- 风险信号识别
- 联系人推荐
- 商机优先级评分
- 下一步动作建议

### 关键点
这里不要让模型“随便想”，而是要有：
- 规则
- 显式评分框架
- 基于证据的推断模板

即：

> LLM 负责归纳和解释，规则负责边界和一致性。

---

## 第 5 层：内容生成层（Communication Generation Layer）

基于上一步已经得出的结论，生成：

- entry point
- wechat_script
- phone_script
- email_script

这一层应当 **最后做**，因为它依赖前面所有分析结果。

而且话术不能脱离证据，否则会显得“空”。

---

## 第 6 层：结果验证与输出层（Validation Layer）

负责：

- 校验输出是否完整
- 校验枚举值是否合法
- 校验关键结论是否有证据引用
- 补默认值 / 降级处理
- 组装最终 JSON

这是保证稳定性的关键。

### 包括
- 去重
- 去广告/去模板内容

- 标准化日期
- 判断是否真的与目标公司相关

### 4.1 企业基本画像抽取

- 企业类型
- 规模
- 主营业务

- 总部地点

### 4.2 近期动态抽取
从新闻、官网新闻、媒体稿里抽：

- 日期
- 标题
- 摘要

---

### 4.3 需求信号候选抽取
从招聘、新闻、产品发布、组织扩张中抽“候选信号”：

比如：

- 新开城市/新业务线
- 推数字化转型项目
- 推出新产品
- 组织架构升级
- 海外扩张
- 融资后进入建设期

这些还不是最终“需求结论”，只是候选线索。

### 4.4 风险信号候选抽取

### 4.5 组织与角色线索抽取

# 六、真正的“智能体推断”应该发生在哪

到了这里，才开始进入你真正想做的“分析智能体”。

我建议分成 5 个推断子任务。

---

## 1. 企业画像归纳器

### 输入
- 基本资料事实
- 官网内容
- 产品/行业描述

### 输出
- `company_profile`

### 关键要求
- 尽量基于高置信度来源
- 不确定字段允许空，不要编造
- 行业、规模、业务要统一口径

---

## 2. 需求信号分析器

### 输入
- 招聘线索
- 扩张事件
- 产品动态
- 数字化信息
- 我方产品描述
- ICP 信息

### 输出
- `demand_signals`

### 核心逻辑
这一步不是“发现事实”，而是把事实转成商务含义。

例如：

- 招聘大量销售运营 → 可能存在销售流程管理需求
- 招聘实施/客户成功 → 可能正在扩大客户交付体系
- 扩张多个区域 → 可能需要统一客户管理和协同系统
- 融资后增长期 → 更可能采购中后台系统

### 这里非常适合“规则 + LLM”
例如：

- 先规则映射出候选需求类型
- 再让 LLM 生成描述、证据、推断解释

这样比纯 Prompt 稳很多。

---

## 3. 联系人推荐器

### 输入
- sales_goal
- 我方产品类型
- 目标角色
- 企业组织线索
- 需求信号类型

### 输出
- `recommended_target_roles`

### 逻辑示例
如果你卖的是 CRM 系统，则推荐角色可能包括：

- 销售负责人
- 销售运营
- 市场负责人
- 客服负责人
- IT/数字化负责人
- 业务条线负责人

### 推荐不能只看产品，还要看场景
比如：

- `first_touch`：优先容易接触、最可能有痛点的人
- `meeting_prep`：优先给出参会相关角色图谱
- `solution_pitch`：优先推荐预算/决策/使用三类角色
- `account_planning`：优先推荐多线程经营路径

---

## 4. 风险分析器

### 输入
- 法务/经营/舆情证据
- 公司动态
- 行业背景

### 输出
- `risk_signals`

### 关键原则
风险模块一定要保守：
- 没证据就不要写
- 有证据但影响不明确，就给低/中
- 风险描述要事实化，不要夸大

因为这部分最容易造成误判。

---

## 5. 商务评分器 / 决策器

这是你整个系统最关键的一层，因为你 README 已经定义了评分模型。

### 输入
- ICP 匹配度
- 需求信号
- 可触达可行性
- 风险扣分

### 输出
- `customer_fit`
- `opportunity_level`
- `priority_score`
- `priority_reason`
- `suggested_next_steps`

### 这里建议：
**评分规则尽量规则化，不要完全交给 LLM。**

因为你已经有：
- 评分维度
- 分数上限
- P1/P2/P3/discard 映射

所以应做成：

1. 规则模块先算基础分
2. LLM 负责解释“为什么得这个分”
3. 最终输出结构化结果

这样可解释性和稳定性都会更好。

# 十、智能体工作流更适合“图”而不是“链”

如果你只做线性流程：

输入 → 抽取 → 分析 → 输出

会太粗。

你这个项目更适合下面这种 DAG 思路：

```text
用户输入
  ↓
任务上下文生成
  ↓
企业身份确认
  ↓
多源信息采集
  ↓
证据清洗/去重/打标签
```

# 十一、你这个项目应该尽量避免的几个坑

---

## 1. 不要把“搜索”和“分析”混在一起
采集阶段负责拿材料；
分析阶段负责判断。  
两者分离，否则后期很难调试。

---

## 2. 不要一开始就追求全自动全网抓取
MVP 可以先只支持：
- 官网
- 新闻
- 招聘
- 基础风险信息

先把流程跑通，比接很多源更重要。

---

## 3. 不要所有字段都让大模型自由生成
尤其以下内容应优先规则化：
- 枚举值
- 分数
- 优先级
- 风险等级
- 字段完整性

---

## 4. 不要没有证据链
你已经设计了 `evidence_references`，这非常重要。  
建议任何关键判断都能追溯到至少一条证据。

---

## 5. 不要只输出最终结果，不保留中间结果
后期一旦结果不对，你会完全不知道错在哪一环。
